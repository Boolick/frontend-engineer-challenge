## [2026-04-01] — Выбор технологического стека

**Задача:** Определить фреймворк и архитектурный подход.

**Предложение агента:** Использовать `Vite + React` — быстрый старт, нулевая конфигурация, отлично подходит для SPA. Агент аргументировал скоростью разработки и простотой деплоя на любой статический хостинг.

**Моё решение:** `Next.js 16 (App Router)`.

**Аргументация:**

В рамках данного ТЗ Vite+React технически достаточен. Однако я сознательно выбрал Next.js, чтобы продемонстрировать инженерное мышление уровня production:

1. **BFF из коробки.** Next.js Route Handlers позволяют реализовать паттерн Backend-For-Frontend без отдельного сервера. Это фундамент всей безопасной работы с токенами (см. ниже).
2. **Безопасность.** Хранение JWT в `localStorage` (единственный вариант для чистого SPA без BFF) — прямая уязвимость к XSS. С Next.js токены живут в `HttpOnly` cookies, созданных на сервере Next.js-слоя, и клиентский JavaScript их никогда не видит.
3. **Производительность и SEO.** Server Components, стриминг, Edge Middleware — не нужны здесь сейчас, но наличие этих инструментов означает, что приложение уже готово к масштабированию без переписывания.
4. **Edge Middleware для auth-guard.** Защита роутов (`/dashboard`) происходит на сетевом уровне до первого байта HTML — без "мигания" экрана, характерного для SPA с client-side redirect.

**Trade-off:** Дополнительная сложность конфигурации (SSR/RSC/Route Handlers), дополнительный сетевой хоп Client → Next.js → Go (≈5-10мс). Для auth-запросов это абсолютно неощутимо.

## [2026-04-02] — Управление состоянием: XState vs Zustand

**Задача:** Реализовать управление состоянием auth-флоу (loading, error, rate limit таймер, success).

**Предложение агента:** Использовать `Zustand` — минималистичный стор, популярный в React-экосистеме, прост в тестировании.

**Моё решение:** `XState v5` (двойная машина: `authMachine` + `errorMachine`).

**Аргументация:**

Auth-флоу с Rate Limiting, таймером обратного отсчёта и множеством edge-cases — это классический **конечный автомат (Finite State Machine)**. Zustand заставляет имитировать его через набор булевых флагов (`isSubmitting`, `isError`, `isRateLimited`, `isSuccess`), что неизбежно ведёт к багам неконсистентности состояний:

```
// Zustand — опасная комбинация:
{ isLoading: true, isError: true } // невозможное состояние, но Zustand это допускает
```

XState делает невозможные состояния **невозможными на уровне типов и архитектуры**:

- `idle` → `submitting` → `authenticated | error | rateLimited` — строго определённые переходы.
- Rate Limit таймер реализован как **внутренний актор** (XState Actor Model) с событиями `TICK` — без `setInterval` в компонентах.
- `errorMachine` — вторая, параллельная машина для глобальных уведомлений. Разделение ответственности: `authMachine` управляет формой, `errorMachine` — глобальной очередью тостов.

**Паттерн:** [Statechart / Hierarchical State Machine](https://statecharts.dev/) — формализация из теории автоматов (Дэвид Харел, 1987), применённая к UI через XState.

**Trade-off:** Более крутая кривая обучения для команды. Однако для запутанного асинхронного флоу с таймерами это осознанная инвестиция в надёжность, а не переусложнение.

## [2026-04-03] — Персистентность Rate Limit: sessionStorage vs Cookie Sync

**Задача:** Обеспечить сохранение таймера обратного отсчёта при обновлении страницы, смене языка и навигации.

**Предложение агента:** Сохранять `retryAfter` timestamp в `sessionStorage` при получении 429, читать при монтировании хука.

**Моё решение:** Синхронизация через **client-accessible cookie**, устанавливаемую на BFF-слое при нормализации ошибки 429.

**Аргументация:**

`sessionStorage` имеет три критических недостатка в данном контексте:

1. **Не переживает смену вкладки** (изолирован на уровне вкладки браузера).
2. **Требует клиентского кода для записи** — значит ошибка 429 должна сначала дойти до компонента, а уже там мы пишем в storage. Это нарушает принцип единственного источника правды.
3. **Не интегрирован с BFF.** BFF уже знает о `retryAfter` при нормализации ошибки (`map-error.ts`) — логично именно там установить куку с `Max-Age`.

**Реализованный подход:**
- `map-error.ts` (BFF): при 429 → `Set-Cookie: auth_rate_limit_expiry=<timestamp_ms>; Max-Age=<retryAfter>; SameSite=Lax`
- `useAuthByEmail.ts` (клиент): при монтировании → `clientCookies.get('auth_rate_limit_expiry')` → вычислить остаток → отправить `RATE_LIMITED` в XState.
- Кука **саморазрушается** по истечению срока — нет ручного cleanup.

**Trade-off:** Кука доступна клиентскому JS (не `HttpOnly`), что сознательно и необходимо по дизайну. Это не чувствительные данные — только timestamp истечения лимита.

## [2026-04-03] — Архитектура проекта: Feature-Sliced Design

**Задача:** Определить структуру папок и модульные границы.

**Предложение агента:** Классическая структура `components/`, `hooks/`, `services/`, `utils/` — проверенная и понятная.

**Моё решение:** **Feature-Sliced Design (FSD)** — `app / widgets / features / entities / shared`.

**Аргументация:**

Стандартная структура не кодирует границы ответственности: любой компонент может импортировать что угодно откуда угодно. FSD решает это через строгую **иерархию слоёв с однонаправленными зависимостями**: `shared` не знает ни о чём выше, `entities` не знают о `features`, и так далее.

Конкретное разделение:
- `entities/session` — `authMachine`, API-методы, типы пользователя.
- `entities/error` — `errorMachine`, типы ошибок, `classifyError()`.
- `features/auth-by-email` — бизнес-сценарий: оркестрация форм + хук `useAuthByEmail`.
- `shared/ui` — UI-kit: `Button`, `FloatingInput` — без бизнес-логики.

## [2026-04-03] — Нормализация ошибок BFF

**Задача:** Как клиент должен получать ошибки от Go-бэкенда?

**Предложение агента:** Обрабатывать HTTP-статусы прямо в компоненте (`if (status === 429)`, `if (status === 401)` и т.д.).

**Моё решение:** Централизованный **BFF Error Normalizer** (`map-error.ts`).

**Аргументация:**

Go-бэкенд возвращает `rpcStatus` формат (gRPC-Gateway). Клиент не должен знать об этих деталях — это утечка деталей реализации бэкенда на фронтенд. Вместо этого:

- `map-error.ts` — единственное место в проекте, знающее о "сыром" формате бэкенда.
- Все Route Handlers вызывают `mapBackendError()` → клиент получает каноническое `{ code, message, retryAfter?, status }`.
- При замене бэкенда нужно будет изменить только один файл.

Это паттерн **Anti-Corruption Layer** из DDD — граница между двумя доменными моделями.

## [2026-04-04–06] — UI-реализация, тесты, i18n, документация

- **Floating Labels:** Анимированные метки через Framer Motion — реализация без JS-библиотек для floating label, только CSS-трансформации + `motion`.
- **4-Layer Error Defense:** Ошибки обрабатываются на 4 уровнях: BFF нормализация → XState errorMachine → Error Router (`classifyError`) → UI (`AuthFeedback` + `ErrorNotificationLayer`).
- **i18n:** `next-intl` — поддержка `ru`/`en` с персистентным переключением через куку. RTL не нужен, но структура `messages/` готова к расширению.
- **Тесты:** Vitest покрывает `authMachine` (все переходы + таймер), `errorMachine` (очередь уведомлений), `proxy.ts` (edge middleware guard).
- **Health Gate:** Формы проверяют доступность бэкенда через `GET /api/auth/health` на маунте и перед каждым submit — блокируют кнопку при недоступности сервиса.
